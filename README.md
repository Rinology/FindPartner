# FindPartner 프로젝트 개요 및 운영 가이드

이 문서는 **FindPartner** 프로젝트의 구조를 분석하고, 데이터 및 코드를 업데이트하는 방법, 그리고 보안을 위해 적용된 reCAPTCHA 기능의 동작 원리를 설명하기 위해 작성되었습니다.

---

## 1. 프로젝트 분석 (Architecture)

FindPartner는 전국 대리점(협력점)의 위치를 지도 상에 표시하고, 우수협력점/원케어/시승가능 여부 등을 필터링하여 사용자에게 제공하는 웹 애플리케이션입니다.
기존 Vanilla JS 환경에서 **React + Vite + TailwindCSS** 기반으로 성공적으로 마이그레이션 되었습니다.

* **프론트엔드 (Frontend):** 
  * **React** 컴포넌트 구조로 분할되어 데이터 상태에 따라 동적으로 UI를 렌더링합니다 (`src/`).
  * 스타일링은 **TailwindCSS**를 활용하여 빠르고 일관되게 관리합니다.
  * 지도 렌더링을 위해 **Leaflet.js** 라이브러리와 OpenStreetMap/CartoDB 타일을 사용하며, React 사이드이펙트(`useEffect`)로 안전하게 캡슐화되어 있습니다 (`src/components/MapPanel.jsx`).
* **전역 상태 관리 (State):** 
  * Context API (`src/StoreContext.jsx`)를 사용하여 검색, 필터링, 데이터 로딩 상태 등을 통합 관리합니다.
* **백엔드 (Backend):** 
  * Vercel의 **Serverless Functions** (`api/gas-proxy.js`)를 사용하여 API 프록시 서버 역할을 합니다.
* **데이터베이스 (Database):** 
  * 별도의 DB 서버 없이 **Google Apps Script (GAS)**와 연결된 Google SpreadSheet를 데이터 소스로 활용합니다.
* **보안 및 최적화 (Security & Performance):** 
  * **Upstash Redis**를 활용하여 트래픽 제한(Rate Limiting) 및 데이터 5분 캐싱(Caching)을 수행합니다.
  * 봇의 무분별한 API 요청을 막기 위해 **Google reCAPTCHA v3**가 적용되어 있습니다.

---

## 2. 로컬 실행 및 테스트 가이드 (Development)

로컬 환경에서 프로젝트를 실행하고 테스트하는 방법입니다.

### 2.1 개발 서버 실행 (Localhost:3000)
본 프로젝트는 `/api` 경로의 서버리스 함수와 프론트엔드가 동시에 실행되어야 정상적으로 작동합니다.
따라서 로컬 테스트 시에는 단순히 Vite 서버만 띄우는 것이 아니라 **Vercel CLI를 사용하여 두 환경을 동시에 구동**해야 합니다.

```bash
# Vercel 개발 서버 실행 (권장)
npm run start-dev
```
위 명령어를 실행하면 `localhost:3000`에서 로컬 서버가 시작되며, reCAPTCHA 및 API 연동이 실제 서버와 동일하게 정상적으로 작동합니다.

*(참고: `npm run dev` 명령어는 단순히 프론트엔드 Vite 서버만 5173 포트로 열기 때문에 CORS 또는 API 에러가 발생할 수 있습니다.)*

## 3. 업데이트 방법 (How to Update)

프로젝트의 업데이트는 크게 '데이터 업데이트'와 '코드/기능 업데이트' 두 가지로 나뉩니다.

### 3.1 매장 데이터 업데이트 (Data Update)
매장 정보(이름, 주소, 연락처, 등급, 휴무일 등)를 수정하거나 추가할 때는 코드를 수정할 필요가 없습니다.
1. 연결된 **Google Spreadsheet** 문서를 열어 데이터를 수정합니다.
2. Vercel 서버(Redis)가 5분(300초) 단위로 캐시를 유지하므로, **스프레드시트를 수정한 뒤 약 5분이 지나면 웹사이트에 자동으로 최신 데이터가 반영**됩니다.

### 3.2 코드 및 설정 업데이트 (Code Update)
UI를 변경하거나 새로운 기능을 추가할 때는 로컬 환경에서 코드를 수정한 뒤 GitHub에 푸시해야 합니다.
1. **환경 설정 변경:** API 주소, 네이버/카카오 외부 링크, reCAPTCHA Site Key 등은 `src/config.js` 파일에서 통합 관리합니다.
2. **UI/로직 변경:** `src/` 폴더 내의 React 컴포넌트를 수정합니다.
3. **배포 (Deployment):** 코드를 수정한 후 GitHub 저장소의 `main` 브랜치로 `git push`를 수행하면, **Vercel을 통해 자동으로 빌드 및 배포**가 진행됩니다. (빌드 명령어: `npm run build`)
4. **환경 변수 (Secret Keys):** 백엔드에서 사용하는 민감한 키(예: `RECAPTCHA_SECRET_KEY`, `GAS_URL`, `UPSTASH` 관련 키)는 코드에 직접 하드코딩하지 않고, **Vercel 대시보드(Settings > Environment Variables)**에서 관리하고 업데이트해야 합니다.

## 4. reCAPTCHA 보안 기능 동작 원리

이 프로젝트에는 경쟁사나 악의적인 봇이 매장 데이터를 무단으로 대량 수집(Scraping)하거나 API 서버를 공격하는 것을 막기 위해 **Google reCAPTCHA v3**가 연동되어 있습니다. 사용자 편의성과 보안성을 동시에 챙기도록 설계되었습니다.

### 4.1 작동 흐름 (Workflow)
1. **투명한 검증:** 사용자가 페이지에 접속하면, 화면에 귀찮은 신호등/횡단보도 찾기 팝업을 띄우지 않고 백그라운드에서 조용히 사람임을 증명하는 '토큰'을 발급받습니다 (`grecaptcha.execute`).
2. **데이터 요청:** 프론트엔드는 발급받은 토큰을 백엔드(`api/gas-proxy.js`)로 보냅니다.
3. **엄격한 서버 검증:** 백엔드는 구글 서버에 해당 토큰이 유효한지 묻고, **"봇 점수(Score)가 0.5점 미만"**이거나 **"토큰이 아예 없는 경우"** 즉시 데이터를 차단하고 `403 Forbidden` 에러를 반환합니다.

### 4.2 프론트엔드의 유연한 예외 처리 (Fail-Safe)
* **광고 차단기 문제 해결:** 일부 사용자들은 브라우저 광고 차단기(Adblock) 때문에 구글 reCAPTCHA 스크립트 자체가 막혀 토큰을 발급받지 못하는 경우가 발생합니다.
* **조용한 우회 (Bypass):** 이때 웹사이트가 멈춰버리는 것을 방지하기 위해, 프론트엔드 코드에서는 일단 에러를 삼키고 **토큰 없이 데이터를 요청**하도록 우회(Bypass) 처리를 해두었습니다.
* **안전한 거절 안내:** 하지만 백엔드(서버)에는 보안 설정이 강력하게 적용되어 있어 토큰이 없는 요청은 거절(403 에러)합니다. 프론트엔드는 이 거절 응답을 받으면 웹사이트를 먹통으로 만드는 대신, 리스트 화면에 **"데이터 로드 실패"**라는 문구를 띄워 사용자에게 현재 네트워크 통신에 문제가 있음을 시각적으로 안전하게 안내합니다.

**결론:** 프론트엔드는 사용자의 화면이 깨지지 않게 유연하게 대처하고, 백엔드는 원칙대로 봇을 완벽하게 차단하는 구조로 완성되어 있습니다.
